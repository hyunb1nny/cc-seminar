토큰이란 무엇인가

Claude가 텍스트를 처리하는 단위 · 한국어 vs 영어 차이

토큰 기초

Claude가 텍스트를 읽고 쓸 때 사용하는 기본 단위입니다. 1 토큰 ≈ 0.75 영어 단어. 모든 비용은 이 토큰 개수를 기준으로 산정됩니다.

0.75
영어 단어/토큰
0.5
한국어 단어/토큰
2–4x
한국어 오버헤드

Input vs Output 토큰

종류정의일반 비율
Input 토큰Claude에게 보내는 것: 내 메시지 + 대화 히스토리 + CLAUDE.md + 파일 내용 등약 80–90%
Output 토큰Claude가 돌려주는 것: 응답 텍스트 + 생성 코드 등약 10–20%
Input 토큰이 훨씬 많지만 Output 토큰이 더 비쌉니다 — 보통 Input의 3–5배. 긴 응답을 유발하는 모호한 요청은 이중 손해입니다 (큰 Input + 긴 Output).

비용 구조

Claude Code를 CLI로 사용하면 내부적으로 Anthropic API를 호출합니다. 요금은 입력/출력 토큰 개수 × 단가로 산정됩니다.

예시 계산 — Opus로 파일 10개 분석 시
  • CLAUDE.md 500 토큰
  • 각 파일 평균 1,000 토큰 × 10개 = 10,000 토큰
  • 대화 히스토리 5회 × 500 토큰 = 2,500 토큰
  • Input 총계 ≈ 13,000 토큰 (+ 이전 대화 누적)
  • 응답 1,500 토큰 × Opus 단가 = 상당한 비용

이 작업을 Haiku로 하면 같은 요청이 약 50배 저렴합니다. 모델 선택이 핵심입니다.

한국어 vs 영어 — 토큰 효율 차이

한국어는 영어보다 토큰을 더 많이 씁니다. 이건 Claude Code만의 문제가 아니라 LLM 전반의 특성입니다.

텍스트예시토큰 수 (근사치)
영어 단어"This is a test message"≈ 5 토큰
한국어 음절"이것은 테스트 메시지입니다"≈ 10–15 토큰
한국어 전문 용어"저장장치 입출력 성능 분석"≈ 15–20 토큰

실무 시사점

  • CLAUDE.md는 한국어로 써도 괜찮습니다 — 캐싱 대상이라 반복 비용은 낮음
  • Skills 설명문(description)이 한국어면 매 호출마다 약간 더 소비됨. 영어도 고려.
  • 로그, 코드, 명령어는 언어 차이 없음 — ASCII 위주라 토큰 비율 동일
  • 코드/로그 분석은 한국어 오버헤드 영향이 작습니다. CLAUDE.md, Skills 파일 쪽이 더 큰 영향을 줍니다.

컨텍스트가 비용에 미치는 영향

어디서 토큰이 새는지 파악하고 진단하기

어디서 토큰이 소비되나?

대화 중 Claude의 컨텍스트 윈도우에 누적되는 것들입니다. 각각이 매 요청마다 다시 전송됩니다 — Claude는 상태를 유지하지 않기 때문입니다.

① 대화 히스토리 50%+
대화가 길어질수록 기하급수적으로 증가. 가장 큰 주범. 10회 왕복이면 이전 모든 대화가 매번 재전송됩니다.
② 도구 실행 결과 20–40%
Bash 출력 100줄, 대용량 파일 Read 등. 특히 Python 분석 결과 전체, 로그 전체 읽기가 한 번에 수만 토큰을 소비할 수 있습니다.
③ CLAUDE.md 5–10%
매 대화 자동 로드. 그러나 캐싱 대상이라 두 번째 요청부터는 비용이 90% 줄어듭니다. 긴 CLAUDE.md도 첫 요청 빼면 거의 공짜.
④ MCP 도구 이름 ~120 토큰
지연 로딩(deferred) 기본값: 도구 이름만 로드, 스키마는 실제 사용 시 로드. 안 쓰는 서버는 비활성화 권장.
결론: 절약의 핵심은 ① 대화 히스토리 관리와 ② 도구 출력 최소화입니다. CLAUDE.md는 캐싱이 되므로 길이보다 내용의 질이 중요합니다.

/context 진단 — 어디를 줄여야 하는지 파악

느려졌거나 비용이 걱정될 때 어디가 부풀었는지 파악하는 명령어들입니다.

/usage   # 현재 대화의 토큰·비용 현황
/context # 카테고리별 컨텍스트 점유 현황 (시각적)

/context를 실행하면 어떤 부분이 컨텍스트의 얼마나 차지하는지 카테고리별로 보여줍니다. 파일 읽기 결과가 크면 Grep으로 전환, 대화가 길면 /compact나 /clear를 고려합니다.

적신호 신호들

  • 응답 속도가 눈에 띄게 느려짐 (컨텍스트 포화)
  • Claude가 이전 내용을 자꾸 잊거나 혼동함 (컨텍스트 초과 위험)
  • 같은 분석인데 갈수록 비용이 증가 (히스토리 누적)
  • 긴 파일 Read 이후 급격한 속도 저하

모델 선택 & 프롬프트 캐싱

같은 작업을 최적 모델로 · 반복 비용을 자동 절감하는 캐싱

모델 선택 — 전략의 50%

같은 작업을 어떤 모델로 하느냐에 따라 비용이 수십 배 차이납니다. 항상 Opus를 쓸 이유가 없습니다.

모델컨텍스트특성적합 작업
Haiku 4.5 200K 토큰 가장 빠르고 저렴 (~50배 ↓) 라우팅, 분류, 간단 요약, 테스트 초안
Sonnet 4.6 1M 토큰 중간 · 성능/비용 최적 일상적 코딩, 분석, 리포트 생성
Opus 4.8 1M 토큰 최고 성능 복잡한 추론, 설계 결정, 아키텍처 검토

선택 가이드

Haiku로 충분한 경우

  • 단순 포맷 변환
  • 파일명 분류·정렬
  • 기계적 템플릿 채우기
  • 단순 CRUD 코드

Sonnet이 적합한 경우

  • 일상적 코드 작성
  • 로그 분석 및 리포트
  • 테스트 작성
  • 중간 규모 리팩토링

Opus가 필요한 경우

  • 복잡한 버그 추론
  • 아키텍처 설계 결정
  • 모호한 요구사항 분석
  • 새 기술 도입 검토

모델 전환 방법

# CLI에서 실시간 전환
/model haiku      # Haiku 4.5로 전환
/model sonnet     # Sonnet 4.6으로 전환 (기본값)
/model opus       # Opus 4.8로 전환

# 설정 파일로 기본값 지정
// .claude/settings.json
{ "model": "claude-sonnet-4-6" }

# 특정 작업만 Haiku로
/model haiku
"이 CSV 파일들 날짜 형식 일괄 변환해줘"
/model sonnet  # 작업 후 다시 기본으로
실전 권장: 기본 모델을 Sonnet으로 설정하고, 복잡한 설계 결정 시만 일시적으로 Opus로 전환. 간단한 반복 작업은 Haiku로 내려 5–50배 절감.

프롬프트 캐싱 — 반복 비용 자동 절감

컨텍스트 윈도우의 앞부분이 이전 요청과 동일하면 다시 처리하지 않고 캐시에서 재사용합니다. 이를 Prompt Caching이라 합니다.

캐시 조건
  • 1,024 토큰 이상의 공통 앞부분
  • 이전 요청과 완전히 동일한 순서로 등장
  • 캐시는 자동 적용 — 별도 설정 불필요
절감 효과
  • 캐시 히트 시 Input 비용 90% 절감
  • 캐시된 토큰은 처리 속도도 빨라짐
  • API 사용: TTL 5분, 구독 사용: 1시간

무엇이 캐시되나?

항목캐싱 여부이유
CLAUDE.md✓ 캐시됨대화 시작마다 동일한 내용이 앞부분에 등장
Skills 파일 내용✓ 캐시됨Skills 호출 시 동일한 파일 내용 반복
대화 히스토리✗ 매번 새로매 턴마다 내용이 추가되어 바뀜
도구 실행 결과✗ 매번 새로실행할 때마다 다른 결과
CLAUDE.md를 길게 써도 처음 한 번 이후는 캐시에서 재사용되므로 비용 부담이 거의 없습니다. CLAUDE.md를 풍부하게 유지하는 것이 절약보다 효과적입니다.

API와 구독의 TTL 차이

API 키로 직접 호출하면 캐시 TTL이 5분입니다. Claude Code 구독 계정에서는 1시간 자동 캐싱이 적용되어 장기 작업에서도 캐시 혜택을 받을 수 있습니다.

절약 전략 — 기초

즉시 적용 가능한 7가지 실전 전략

/rewind — 잘못된 방향에서 즉시 되돌아가기

대화가 잘못된 방향으로 흘렀거나 응답이 마음에 들지 않을 때, 이미 소비한 토큰을 되살릴 순 없지만 그 시점으로 돌아가서 다시 시작할 수 있습니다.

동작 방식
  1. /rewind 입력 → 이전 대화 지점 선택
  2. 선택한 지점 이후의 모든 대화와 도구 실행이 히스토리에서 제거됨
  3. 해당 시점부터 새로운 방향으로 재시작
✗ 비효율적인 방법

나쁜 응답 후 계속 "아니야", "다시 해줘"를
반복하며 수정 요청.

→ 잘못된 응답들이 히스토리에 축적
→ 매 요청마다 그 쓰레기도 함께 전송
→ 갈수록 느려지고 비용 증가
→
✓ /rewind 사용

나쁜 응답 직후 /rewind
→ 그 요청 이전 시점 선택
→ 더 명확하게 다시 요청

히스토리 오염 없이 깔끔하게
다른 방향으로 재시작.
/branch와 차이: /rewind는 현재 대화를 해당 시점으로 되돌립니다. /branch는 현재 대화를 유지하면서 새 분기를 만듭니다. "이 방향도 해보고 싶다"면 /branch, "이 방향은 버리겠다"면 /rewind.

/clear · /compact — 컨텍스트 관리

누적된 대화 히스토리가 속도 저하와 비용 증가의 가장 큰 원인입니다. 상황에 맞게 두 가지 도구를 씁니다.

/clear — 전체 초기화

  • 대화 히스토리를 완전히 삭제
  • CLAUDE.md와 설정은 유지됨
  • 다음 대화는 완전한 새 시작

언제: 다른 주제로 전환 시. 로그 분석이 끝나고 코드 작업으로 넘어갈 때.

/compact — 압축 요약

  • 히스토리를 요약으로 압축
  • 핵심 맥락은 유지, 부피만 줄임
  • 같은 주제의 긴 작업 중간에 사용

언제: 같은 모듈 작업이 길어졌을 때. 히스토리는 유지하되 비용 줄이고 싶을 때.

실전 사용 패턴

# 패턴 1: 작업 단위로 /clear
[로그 분석 완료]
/clear
[코드 수정 시작]
"parse.py 에서 응답시간 계산 로직 수정해줘"

# 패턴 2: 장시간 작업에서 /compact
[1시간 이상 작업 후 느려짐을 느꼈을 때]
/compact
"아까 하던 리팩토링 계속해줘"

/handoff — 맥락 유지하며 세션 이전

컨텍스트가 꽉 찼지만 작업을 계속해야 할 때, 현재까지 한 것을 정리해서 새 세션에 넘깁니다.

/handoff

# Claude가 현재 작업 상태를 정리:
# - 완료된 것
# - 진행 중인 것
# - 다음에 해야 할 것
# - 중요한 결정 사항
# → 새 세션에서 이 내용을 첫 메시지로 붙여넣기
/handoff는 mattpocock/skills에 포함된 Skill입니다. 프로젝트 스코프 또는 글로벌 스코프로 설치해서 사용합니다.

vs. /compact

/compact/handoff
세션현재 세션 유지새 세션으로 이전
히스토리요약으로 압축제거, 새로 시작
사용 시점같은 세션 계속, 히스토리만 줄임세션 컨텍스트가 꽉 찼을 때

출력 토큰 줄이기 — 간결한 요청의 힘

Output 토큰이 Input보다 비쌉니다 (3–5배). 응답 길이를 의도적으로 제어하면 실질적인 절감이 됩니다.

직접 길이 지정

# ✗ 모호한 요청 → 장황한 응답
"이 API 로그 분석해줘"

# ✓ 명확한 제약 → 필요한 것만
"API 응답시간 평균값만 ms 단위로 한 줄로"
"에러 유형 TOP 3만 표로"
"결론 2문장으로"

형식 지정으로 통제

# 표로 → 설명 없이 숫자만
"| 엔드포인트 | 응답시간 | 판정 | 형식으로만 출력"

# JSON으로 → 파싱 가능, 설명 제거
"결과를 JSON으로만. 설명 없이"
'{"endpoint": "...", "avg_ms": ..., "status": "ok|warn|fail"}'

Caveman 방식

매번 이렇게 요청하는 게 번거로우면 Caveman Skill을 설치하면 됩니다. /caveman 한 번으로 Claude가 자동으로 간결하게 응답합니다. (Part 6에서 자세히 다룸)

CLAUDE.md 다이어트 — 밀도 높이기

CLAUDE.md는 캐싱 덕분에 길어도 반복 비용이 낮지만, 첫 로드 비용과 Claude의 이해 품질을 위해 간결하게 유지하는 것이 좋습니다.

✗ 과도한 CLAUDE.md

- 모든 파일 목록 열거
- 코드 예시 수십 줄
- 역사/배경/고민 과정
- 이미 코드에 있는 내용 반복
- 안 쓰는 규칙 나열

→ 길지만 신호 대비 노이즈가 큼
→ 첫 대화 비용 높음
→
✓ 밀도 높은 CLAUDE.md

- 폴더 구조 + 역할 (핵심만)
- 자주 쓰는 명령어
- 도메인 약어/용어 정의
- 주의사항 (하지 말 것)

→ 모든 줄이 실제로 쓰임
→ Claude가 더 정확히 이해

제거해도 되는 내용

  • 코드에서 바로 알 수 있는 구조 설명 (Glob으로 탐색 가능)
  • 3개월 이상 바뀐 적 없는 절차 (지금도 유효한지 검토)
  • 팀원 이름, 담당자 정보 (Memory에 넣는 것이 적합)
  • 전체 API 명세 (URL 링크로 대체하거나 subagent에게 읽히기)

파일 제외 설정 — 불필요한 토큰 차단

Claude에게 읽히지 않아도 되는 파일들이 있습니다. 이를 명시적으로 제외하면 탐색 비용과 실수로 읽을 가능성을 줄입니다.

# .claude/settings.json
{
  "ignore": [
    "node_modules/**",
    "dist/**",
    "*.log",
    "**/__pycache__/**",
    "*.pyc",
    "vendor/**",
    "*.min.js"
  ]
}
vendor 코드란? 서드파티 라이브러리 소스를 패키지 매니저 없이 프로젝트 디렉토리 안에 직접 복사해 둔 것입니다. node_modules/, vendor/, 빌드 산출물 등이 여기에 해당합니다. Claude가 이걸 읽으면 수십만 토큰이 순식간에 소진됩니다.

.gitignore에 있는 파일들은 기본적으로 Claude도 무시하지만, 명시적으로 지정하는 것이 더 안전합니다.

PDF → Markdown 변환 — 바이너리 비용 제거

PDF와 PPTX는 압축된 바이너리 포맷이라 Claude가 효율적으로 읽기 어렵습니다.

✗

Grep 불가

압축된 바이너리라 텍스트 검색이 안 됨. 줄 기반 offset/limit도 불가.

⚠️

페이지 단위만

pages="1-5"로만 범위 지정 가능. 최대 20페이지/요청. 줄 단위 범위는 사용 불가.

✓

MD 변환 후

Markdown으로 변환하면 Grep, offset/limit 전부 사용 가능. 비용도 30–50% 절감.

markitdown — Microsoft 오픈소스

PDF, PPTX, Word, Excel 등 다양한 포맷을 Markdown으로 변환하는 CLI 도구입니다.

# 설치
pip install markitdown

# 사용
markitdown spec.pdf > spec.md
markitdown report.pptx > report.md
markitdown table.xlsx > table.md

# 이후 효율적으로 읽기
Read("spec.md", offset=0, limit=100)
Grep("응답시간", path="spec.md")
변환된 Markdown 파일은 git에 함께 커밋하면 팀 전체가 Claude에게 효율적으로 스펙을 참조시킬 수 있습니다.

도구 선택의 원칙 · Subagent 활용

CLI vs MCP — 도구 선택의 원칙

같은 작업도 CLI(Bash)로 하느냐, MCP로 하느냐에 따라 토큰 효율이 달라집니다.

CLI가 유리한 경우

  • 로컬 파일 처리 (grep, awk, jq)
  • 빌드/테스트/스크립트 실행
  • 간단한 HTTP 호출 (curl)
  • 파이프 조합으로 전처리

결과를 grep·head·jq 등으로 사전 필터링하면 Claude가 받는 토큰이 줄어듭니다.

MCP가 유리한 경우

  • DB 쿼리 (MongoDB, SQL)
  • 외부 API 인증이 필요한 경우
  • 구조화된 데이터 조회
  • Claude가 스키마를 알아야 할 때

MCP는 지연 로딩으로 미사용 시 오버헤드가 낮습니다. 단, 사용 중에는 스키마가 컨텍스트에 포함됩니다.

실전 패턴 — CLI 전처리 + Claude 분석

# ✗ 거대 출력 그대로 전달
$ python analyze.py logs/ # 출력 3,000줄

# ✓ 사전 필터링 후 핵심만 전달
$ python analyze.py logs/ | grep "ERROR" | tail -20
# → Claude에게 전달되는 토큰: 20줄만

# ✓ jq로 필요한 필드만 추출
$ cat result.json | jq '{avg_ms: .metrics.avg, errors: .errors[:5]}'
# → 전체 JSON 대신 필요한 것만
Claude에게 "이 명령 실행해줘"라고 하기 전에, 출력이 얼마나 될지 먼저 생각하세요. 수천 줄이 예상되면 | head -50 또는 | grep 패턴을 함께 요청합니다.

Subagent — 독립 컨텍스트로 대용량 분석

메인 세션 컨텍스트를 오염시키지 않고 대용량 파일을 분석하는 전략입니다. Subagent는 독립된 컨텍스트 창에서 실행되어 결과만 메인 세션으로 반환합니다.

핵심 패턴:
대용량 분석 → Subagent에게 위임 → 요약만 받기
메인 세션에서 직접 하면 모든 파일 내용이 메인 컨텍스트에 누적됩니다. 
# 메인 세션에서 요청
"지난 7일치 각 API 엔드포인트 로그를 Subagent로 병렬 분석해줘.
각 엔드포인트별 응답시간 평균 · 에러율만 반환해줘."

→ Claude가 내부적으로:
  - Subagent 1: GET /users 로그 분석
  - Subagent 2: POST /data 로그 분석
  - Subagent 3: PUT /config 로그 분석
  (각각 독립 컨텍스트, 병렬 실행)
→ 결과: 3개 요약만 메인 세션으로 반환
→ 메인 세션에 누적된 것: 요약 3개 ≈ 1,000 토큰
  (직접 했다면: 로그 전체 ≈ 100,000+ 토큰)
Subagent는 메인 세션 Memory에 접근하지 않습니다. 분석에 필요한 컨텍스트(규칙, 기준값 등)는 요청 메시지에 명시해서 넘겨야 합니다.

Caveman · Graphify · RTK — 오픈소스 절약 도구

Caveman — 출력 토큰 65% 절감

Claude의 응답을 원시적으로 압축하는 Skills 컬렉션입니다. 장황한 설명을 제거하고 핵심만 전달합니다. 코드와 명령어 정확도는 그대로 유지하면서 설명만 압축합니다.

65%
출력 토큰 절감
~3x
응답 속도 향상
100%
코드 정확도 유지

실제 압축 예시

일반 응답 (긴 설명)

"parse_response 함수를 수정했습니다. 이전에는 정수를 반환했지만 이제 부동소수점을 사용합니다. 이렇게 하면 응답시간 계산의 정밀도가 향상됩니다. 또한 엣지 케이스를 처리하기 위해..."

≈ 80 토큰

Caveman 응답 (압축)

parse_response: int→float. 정밀도↑. 엣지케이스 처리됨.

≈ 18 토큰

한국어 예시

일반 응답

NVMe SSD 컨트롤러 검증 완료. 기록 속도 7,000 MB/s 달성. 읽기 성능 6,850 MB/s. 불량 블록 없음. 온도 정상 범위. 합격 판정.

≈ 62 토큰

Caveman 응답

NVMe검증: W7000/R6850 MB/s. 불량블록0. 온도정상. 합격✓

≈ 18 토큰 (71% 절감)

설치 및 명령어

# 설치
curl -fsSL https://raw.githubusercontent.com/JuliusBrussee/caveman/main/install.sh | bash
# 또는
git clone https://github.com/juliusbrussee/caveman
cp caveman/*.md ~/.claude/skills/

# 사용
/caveman           # 기본 압축
/caveman lite      # 약하게
/caveman full      # 강하게
/caveman ultra     # 극단적 압축
/caveman-commit    # 50자 커밋 메시지
/caveman-stats     # 절감된 토큰·비용 확인
코드 리뷰나 설계 논의는 일반 모드, 반복 실행 스크립트나 로그 분석 결과는 Caveman 모드. 항상 ultra로 쓰지 말고 목적에 맞게 강도를 선택하세요.

Graphify — 코드베이스 지식 그래프화

프로젝트를 지식 그래프(knowledge graph)로 변환해서 Claude가 코드베이스 탐색에 쓰는 토큰을 줄입니다. Claude가 "어디가 어디에 연결되어 있는지"를 탐색하는 과정을 미리 저장해두는 방식입니다.

동작 방식

  • PreToolUse Hook에 등록해 Claude가 파일을 읽기 전에 자동 실행
  • 파일 간 의존성, 함수 호출 관계를 그래프로 캐싱
  • Claude가 특정 파일이나 함수를 찾을 때 전체 Glob/Read 대신 그래프 조회
💡 실무 활용

큰 모노레포나 의존성이 복잡한 프로젝트에서 효과적입니다. 파일 수가 수십 개 이하인 작은 프로젝트는 Graphify 없이도 충분합니다.

설치

# 프로젝트에서
git clone https://github.com/yannbam/graphify
cp graphify/graphify.md .claude/skills/

# PreToolUse Hook에 등록 (.claude/settings.json)
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Read|Glob",
      "hooks": [{
        "type": "command",
        "command": "python graphify/update_graph.py"
      }]
    }]
  }
}

RTK (Rust Token Killer) — Bash 출력 80% 압축

Claude에게 전달되는 Bash 도구 출력을 80% 압축하는 Rust 기반 CLI 도구입니다. Claude가 Bash 명령 결과를 받을 때, RTK가 중간에서 가로채 요약해서 전달합니다.

80%
Bash 출력 압축
Rust
구현 언어 (빠름)

동작 원리

# 일반 Bash 도구 흐름
Claude ──── Bash 명령 실행 ──── 3,000줄 출력 ──→ Claude

# RTK 적용 후
Claude ──── Bash 명령 실행 ──── 3,000줄 출력
                                     ↓ RTK 압축
                               600줄 요약 ──→ Claude

설치

# Cargo(Rust)로 설치
cargo install rtk

# 또는 prebuilt 바이너리
curl -fsSL https://github.com/yourusername/rtk/install.sh | bash

# Claude Code의 Bash 출력에 자동 적용
// .claude/settings.json
{
  "bash": { "compressor": "rtk" }
}
주의: RTK는 출력을 자동으로 요약합니다. 세부 출력이 중요한 경우(에러 메시지 정확한 내용, 특정 줄 번호 등) RTK가 압축하면서 중요 정보를 누락할 수 있습니다. 디버깅 시에는 RTK를 임시 비활성화하는 것을 고려하세요.

RTK vs. 수동 필터링

RTK (자동)
  • 설정 후 신경 쓸 필요 없음
  • 모든 Bash 출력에 자동 적용
  • 요약 품질이 일정하지 않을 수 있음
수동 필터링 (grep/head/jq)
  • 매번 명시해야 함
  • 정확히 필요한 것만 추출
  • 중요 정보 누락 없음

핵심 정리 — 비용 절감 체크리스트

1️⃣
모델을 맞게 선택

기본 Sonnet, 복잡한 설계만 Opus, 단순 반복은 Haiku. 이것만으로 최대 50배 절감.

2️⃣
작업 단위로 /clear, 중간에 /compact

대화 히스토리가 50%+를 차지. 관련 없는 작업 전환 시 반드시 /clear.

3️⃣
파일 전체 읽기 대신 범위 지정

Read + offset/limit, 또는 Grep으로 필요한 줄만. PDF는 pages 파라미터 활용.

4️⃣
Bash 출력 사전 필터링

명령 결과를 Claude에게 넘기기 전에 grep·head·jq로 핵심만 추출.

5️⃣
PDF/PPTX → Markdown 변환

markitdown으로 변환 후 관리. Grep 사용 가능 + 비용 30–50% 절감.

6️⃣
구체적인 요청으로 Output 통제

"한 줄로", "표로만", "JSON으로만" → 응답 길이 제어 → Output 토큰 절감.

7️⃣
커뮤니티 도구 활용

Caveman(출력 65%↓) · RTK(Bash 출력 80%↓) · Graphify(탐색 비용↓)

우선순위

효과 순위: 모델 선택 > 히스토리 관리(/clear/compact) > 도구 출력 최소화 > Caveman/RTK
모델 선택 하나가 다른 모든 전략을 합친 것보다 클 수 있습니다.
