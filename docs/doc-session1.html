AI 활용의 진화

Claude를 잘 쓴다는 것의 의미가 바뀌고 있습니다. 단순히 좋은 프롬프트를 쓰는 것을 넘어, 도구를 활용하고, 컨텍스트를 설계하고, 환경을 자동화하는 단계로 진화하고 있습니다.

PHASE 1

Prompt Engineering

좋은 프롬프트를 작성하면 좋은 답이 나온다. One-shot 질문, 역할 지정, few-shot 예시 등으로 응답 품질을 높이는 단계.

한계: 매번 처음부터. 프로젝트 맥락 없음.

PHASE 2

Context Engineering

프롬프트 하나가 아니라 컨텍스트 윈도우 전체를 설계한다. 무엇을 넣을지, 언제 초기화할지를 의도적으로 관리한다.

이 세미나의 핵심 — Part 2에서 다룸.

PHASE 3

Harness Engineering

Claude가 작동하는 환경 전체를 시스템으로 구성한다. Hooks, Skills, Memory로 팀 전체가 동일한 환경을 공유하게 만든다.

Part 3에서 다룸 — 모든 것의 통합.

오늘 세미나는 이 흐름을 따릅니다: 도구를 알고(Part 1) → 컨텍스트를 설계하고(Part 2) → 시스템으로 자동화한다(Part 3).

Agent & Tools

Claude Code는 왜 "Agent"인가 · 손을 쓰는 도구들

Claude Code는 왜 "Agent"인가?

일반 LLM은 텍스트를 받아 텍스트를 돌려줍니다. Agent는 목표를 받아 스스로 계획하고, 도구를 선택해 실행하고, 결과를 보고 다시 판단합니다.

💬 일반 LLM (ChatGPT 등)
  • 질문 → 텍스트 답변
  • 파일을 직접 읽지 못함
  • 명령을 직접 실행하지 못함
  • 한 번의 응답으로 끝
🤖 Agent (Claude Code)
  • 목표 → 계획 → 도구 실행
  • 파일을 읽고 수정
  • 명령어를 직접 실행
  • 결과를 보고 다음 행동 결정
"로그에서 에러 패턴 분석해줘"
→ Glob으로 파일 탐색 → Read로 로그 읽기 → Bash로 grep 실행 → 결과 분석 → 리포트 Write
이 모든 과정을 Claude가 스스로 판단하며 진행합니다. 사용자는 목표만 제시하면 됩니다.
각 도구 실행 전 승인 요청이 뜨는 것이 기본값입니다. 자주 쓰는 명령은 settings.json에 등록해 자동 승인할 수 있습니다.

Claude Code의 도구들

⚡ Bash

쉘 명령을 직접 실행합니다. 테스트 스크립트, Python 분석, 시스템 명령, 파이프 조합 등 터미널에서 하는 모든 것을 Claude가 대신할 수 있습니다.

📖 Read

파일 내용을 읽습니다. 텍스트 파일은 줄 기반 offset/limit으로 범위 지정이 가능합니다. PDF는 pages 파라미터로 페이지 범위를 지정합니다(최대 20페이지/요청).

✏️ Edit

파일의 특정 부분만 수정합니다. 지정한 문자열을 찾아 교체하는 diff 방식으로 동작합니다. 실수 가능성이 낮고 효율적입니다.

📝 Write

새 파일을 생성하거나 기존 파일을 전체 덮어씁니다. 리포트, 스크립트, 설정 파일 등을 자동으로 생성할 때 사용합니다.

🔍 Grep

파일 내 텍스트를 검색합니다. 정규식을 지원하며 라인 번호를 반환합니다. 로그에서 ERROR 패턴 추출, 함수 정의 찾기 등에 활용합니다.

📂 Glob / LS

패턴으로 파일을 찾거나 디렉토리 목록을 조회합니다. Claude가 작업 전에 프로젝트 구조를 파악할 때 가장 먼저 사용합니다.

🌐 WebFetch / WebSearch

WebFetch는 특정 URL 내용을 가져옵니다. WebSearch는 실시간 웹 검색으로 학습 데이터에 없는 최신 정보를 조회합니다.

🔌 MCP

Model Context Protocol로 외부 시스템을 연결합니다. DB, 사내 API, Jira/Confluence, 장비 제어 등 거의 모든 시스템을 도구로 만들 수 있습니다.

파일 도구 — Read · Edit · Write · Grep · Glob

Read — 범위 지정 읽기

# 텍스트 파일: 줄 단위 범위 지정
Read("logs/app.log", offset=500, limit=50)

# PDF: pages 파라미터 (최대 20페이지/요청)
Read("spec.pdf", pages="1-5")

Edit — 정밀 수정

Edit("scripts/parse.py",
  old="response_ms / 1000",
  new="response_ms / 1000.0")
Claude는 Edit 전에 반드시 Read로 파일을 먼저 읽습니다. 파일을 읽지 않은 상태에서 Edit을 시도하면 오류가 납니다.

Grep — 코드/로그 검색

# 로그에서 패턴 검색
Grep("ERROR\|WARN", path="logs/")

# 코드에서 함수 찾기
Grep("def parse_response", path="scripts/")
Claude는 보통 Glob → Read → Edit/Write 순서로 작업합니다.
① Glob으로 관련 파일 탐색 → ② Read로 내용 확인 → ③ Edit/Write로 수정/생성

웹 도구 — WebFetch · WebSearch

WebFetch — URL 직접 읽기

# 라이브러리 공식 문서, GitHub 이슈 등을 직접 읽기
WebFetch("https://docs.example.com/api/...")

WebSearch — 실시간 검색

WebSearch("React 19 breaking changes migration")
사내 기밀 정보를 WebSearch 쿼리에 포함하지 마세요. 쿼리는 외부 검색엔진으로 전송됩니다.

Bash — 개발 실전 패턴

테스트 실행 후 즉시 분석

# 요청
"python run_tests.py 실행하고 실패한 테스트만 뽑아줘"

# Claude가 내부적으로 실행
$ python run_tests.py --output-format=json

로그에서 오류 패턴 추출

# 요청
"이번 주 app.log에서 ERROR, WARN 발생 패턴 찾아줘"

# Claude가 내부적으로 실행
$ grep -n "ERROR\|WARN" /var/log/app.log | tail -100

permission 설정 — 자주 쓰는 명령 자동 승인

// .claude/settings.json
{
  "permissions": {
    "allow": ["Bash(python:*)", "Bash(npm:*)", "Bash(grep:*)"]
  }
}
주의: rm -rf, 프로덕션 배포 명령은 allow 목록에 넣지 마세요. 매번 명시적 승인을 받는 것이 안전합니다.

MCP — 외부 시스템 연결

MCP(Model Context Protocol)는 Claude가 외부 시스템을 도구로 쓸 수 있게 하는 표준 프로토콜입니다. DB, API, SaaS 서비스 등 거의 모든 외부 시스템을 Claude의 도구로 등록할 수 있습니다.

등록 방법

// ~/.claude.json — 글로벌 설정
{
  "mcpServers": {
    "mongo": {
      "command": "npx",
      "args": ["-y", "@mcp/server-mongodb"],
      "env": { "MONGODB_URI": "mongodb://..." }
    }
  }
}

실전 예시 — 자연어 → DB 쿼리

# 요청
"지난달 응답시간 가장 높은 엔드포인트 TOP 3 알려줘"

# Claude가 자동 생성·실행하는 MongoDB 쿼리
db.api_logs.aggregate([
  { $group: { _id: "$endpoint", avg_ms: { $avg: "$response_ms" } }},
  { $sort: { avg_ms: -1 }},
  { $limit: 3 }
])
MCP 기본값은 지연 로딩(deferred)으로 세션 시작 시 툴 이름(~120 토큰)만 로딩합니다. 안 쓰는 서버는 등록하지 않는 것이 좋습니다.

CLAUDE.md & Context Engineering

프로젝트를 Claude에게 한 번만 설명하기 · 컨텍스트 윈도우 설계

CLAUDE.md란?

Claude Code가 시작될 때 자동으로 읽는 프로젝트 설명 파일입니다. 프로젝트 루트에 두면 대화마다 자동으로 컨텍스트에 포함됩니다.

  • 매번 "우리 프로젝트는 이런 구조야..."를 반복하지 않아도 됩니다
  • 팀원 모두가 동일한 컨텍스트로 Claude를 사용할 수 있습니다
  • 코드베이스 + 도메인 지식을 한 곳에 집약합니다
  • 하위 폴더에도 추가로 두면 해당 디렉토리 작업 시 자동 로드됩니다
👤 사람이 직접 작성

프로젝트 초기 설정, 컨벤션·규칙 정의, 도메인 지식 입력. 팀이 작성·관리합니다.

🤖 Claude가 직접 수정

"CLAUDE.md에 이 컨벤션 추가해줘" 요청 시, Claude가 스스로 Edit 도구로 업데이트할 수 있습니다.

# CLAUDE.md 예시

## 프로젝트 개요
웹 서비스 모니터링 자동화 도구.

## 폴더 구조

logs/    - 수집된 로그 파일
scripts/ - 분석 파이썬 스크립트
reports/ - 자동 생성 리포트

## 빌드/실행
python run_tests.py --env staging

## 규칙
- main 브랜치에 직접 push 금지
- 프로덕션 DB에 직접 쓰지 말 것

Claude는 인터넷의 방대한 Markdown 문서로 학습됐습니다. Markdown 구조 자체가 Claude에게 의미 신호가 됩니다.

잘 쓴 CLAUDE.md 구조

🗂 프로젝트 개요

무엇을 만드는 곳인지, 핵심 도메인 용어 정의. 팀 도메인 약어를 정의해두면 Claude가 즉시 이해합니다.

📁 코드 구조

폴더별 역할 설명. Claude가 "어디를 건드려야 하는지" 탐색 없이 바로 알게 됩니다.

⚡ 빌드 & 실행

자주 쓰는 명령어. Claude가 직접 실행해서 결과를 확인하는 데 씁니다.

📐 코딩 컨벤션

변수명 규칙, 단위 표기 방식, 코드 스타일. Claude가 생성하는 코드가 팀 스타일을 자동으로 따릅니다.

🚫 하지 말아야 할 것

"프로덕션 DB에 직접 쓰지 말 것", "main에 직접 push 금지" 같은 안전 규칙.

🔗 외부 참고

내부 위키, Jira 링크, 스펙 문서 경로. 파일 전체를 복붙하지 말고 경로만 명시하세요.

Context란 무엇인가?

Claude는 무상태(stateless) 모델입니다. 대화가 끝나면 모든 것을 잊습니다. Claude가 응답을 생성하기 직전, "지금 이 순간 알고 있는 모든 것"이 Context입니다.

컴퓨터의 RAM과 같습니다. 전원을 끄면(=대화 종료) 사라지고, 켜는 동안(=한 대화 안에서)만 유지됩니다.
✓ Context 안에 있으면
  • Claude가 알고 있음
  • 답할 수 있음
  • 실행에 반영됨
✗ Context 밖에 있으면
  • Claude가 모름
  • 추측하거나 틀림
  • 반영되지 않음
Context Engineering = 무엇을 Context에 넣을지 의도적으로 설계하는 것. 좋은 Context가 좋은 응답을 만듭니다.

컨텍스트 윈도우 — 무엇이 들어있나?

구성 요소설명특성
① CLAUDE.md프로젝트 설명 파일자동 로드 · 고정 · 캐시됨
② 대화 히스토리이전 대화 전체누적 · 계속 증가
③ 도구 실행 결과Bash 출력, 파일 읽기 결과매번 새로 포함
④ MCP 툴 이름 목록등록된 MCP 서버의 도구 이름지연 로딩 · ~120 토큰
⑤ 현재 요청지금 내가 입력한 메시지매 응답마다 추가
파일 전체를 읽거나 Bash 출력이 길면 ③이 급격히 팽창합니다.

컨텍스트 관리 명령어

  • /usage — 현재 대화에서 쓴 토큰·비용 확인
  • /context — 카테고리별 컨텍스트 점유 현황
  • /clear — 히스토리 전체 초기화 (CLAUDE.md와 설정은 유지)
  • /compact — 히스토리를 요약으로 압축 (맥락은 유지)

모델별 컨텍스트 한도: Haiku 4.5 → 200K / Sonnet 4.6 · Opus 4.8 → 1M 토큰

Context Engineering — 잘 쓰는 전략

① 안정적 컨텍스트 분리

자주 바뀌지 않는 정보는 CLAUDE.md에 고정합니다.

✗ 매번: "우리 프로젝트는 이런 구조이고…"
✓ CLAUDE.md에 한 번 → 자동 로드

② 필요한 것만 읽기

파일 전체 대신 필요한 범위만 Read합니다.

✗ Read("app.log")
✓ Read("app.log", offset=0, limit=100)

③ 작업 단위로 초기화

무관한 작업 전환 시 /clear, 긴 작업 중간엔 /compact.

로그 분석 끝 → /clear → 코드 작업 시작

④ 구체적인 요청

모호한 요청은 Claude가 길게 응답합니다. 응답이 길면 히스토리가 팽창합니다.

✗ "이 로그 분석해줘"
✓ "API 응답시간만 전주 대비 % 변화로 한 줄 요약"

그래도 반복되는 것이 있다 — Skills의 등장

CLAUDE.md로 프로젝트 지식은 해결됐습니다. 하지만 자주 반복하는 요청 패턴은 여전히 매번 타이핑해야 합니다.

매 스프린트마다 반복하는 요청 예시

"지난주 API 에러 로그 분석해서 에러 유형별로 집계하고, 개선이 필요한 엔드포인트 TOP 3 뽑아줘. 결과는 마크다운 표로."

Skills는 이 반복 프롬프트를 /명령어 하나로 등록합니다. /api-report 하나로 위의 긴 요청을 대체하고, 팀 전체가 같은 방식으로 Claude를 씁니다.

Skills는 단순 단축키가 아닙니다. 긴 절차, 판단 기준, 출력 형식까지 담은 팀의 AI 사용 표준입니다.

Harness Engineering

Claude Code 전체를 하나의 시스템으로 설계하기

Harness Engineering이란?

핵심 관점

Claude Code 자체가 이미 잘 구성된 harness입니다. Anthropic이 Claude LLM 위에 도구, 승인 시스템, 파일 접근, MCP 연결을 얹어 만든 것이 첫 번째 레이어입니다.

우리가 하는 것

Skills — 슬래시 명령어 · 자동 호출

👤 User-invoked

사용자가 직접 슬래시 명령어로 호출합니다.

/api-report logs/jan.json
🤖 Model-invoked

Claude가 대화 맥락을 보고 자동으로 실행합니다. Skill의 description frontmatter를 잘 써야 동작합니다.

적용 범위 (Scope)

🌐 Global

~/.claude/skills/ — 내 모든 프로젝트에서 사용. 개인 공통 명령어.

📁 Project

.claude/skills/ — git에 커밋하면 팀원 모두가 동일한 명령어를 사용합니다.

Skill 파일 예시

Memory — 대화를 넘어서 기억하기

Claude는 기본적으로 대화가 끝나면 모든 것을 잊습니다. Memory 시스템은 중요한 사실을 파일로 영구 저장해서, 다음 대화 시작 때 자동으로 불러옵니다.

# "기억해줘"라고 말하거나 /remember 명령어 사용
"우리 팀은 API 응답시간 기준이 200ms야, 기억해줘"

→ Claude가 .claude/memory/ 에 파일로 저장
→ 다음 대화에서 자동으로 적용
팀 공유: .claude/memory/를 git에 커밋하면 팀원 전체가 같은 컨텍스트를 공유할 수 있습니다.

user — 나에 대한 정보

"백엔드 개발자, Python/TypeScript 사용"
→ 매번 소개 없이 맞춤 응답

project — 현재 작업 상태

"REST API 리팩토링 중, 목표 RPS 10K"
→ 다시 설명 불필요

feedback — 선호/비선호 방식

"타입 반드시 명시, 항상 타입 힌트 포함"
→ 같은 피드백 반복 불필요

reference — 외부 참조

"버그는 Jira DEV 프로젝트에 등록"
→ 어디서 찾아야 하는지 알게 됨

CLAUDE.md와 Memory, 무엇이 다른가?

📄 CLAUDE.md

  • 사람이 직접 작성하는 파일
  • 프로젝트 전체에 대한 고정 정보
  • git에 커밋 → 팀 전체 공유
  • 바뀌지 않는 것: 구조, 컨벤션, 실행 명령
  • 팀이 작성·관리

🧠 Memory

  • Claude가 자동으로 저장하는 파일
  • 나(사용자)에 대한 가변적 정보
  • 대화마다 업데이트됨
  • 바뀌는 것: 현재 작업, 선호 방식, 피드백
  • Claude가 직접 쓰고 수정
한 줄 요약:
CLAUDE.md = "우리 프로젝트는 이런 곳이야" (팀이 Claude에게)
Memory = "너(Claude)가 나에 대해 배운 것" (Claude가 스스로 기록)

Hooks — 이벤트 기반 자동화

Claude가 특정 행동을 할 때 자동으로 실행되는 쉘 스크립트입니다. "파일을 수정하면 자동으로 린터 실행", "Bash 명령 실행 전에 위험 명령 차단" 같은 자동화가 가능합니다.

PreToolUse

도구 실행 직전 발동. exit 1을 반환하면 실행 차단. 위험 명령 사전 차단에 사용합니다.

PostToolUse

도구 실행 직후 발동. 파일 수정 후 자동 테스트 실행, 린트 체크 등에 사용합니다.

// .claude/settings.json — 파일 수정 시 자동 테스트
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "python -m pytest scripts/ -q" }]
    }]
  }
}
Hooks는 Claude의 승인 없이 자동으로 동작합니다. 스크립트에 오류가 있으면 Claude의 모든 작업이 차단될 수 있습니다.

Subagents · Planning Mode

Subagents — 전문화된 병렬 에이전트

복잡한 작업을 여러 전문 에이전트에 병렬로 위임합니다. 각 에이전트는 독립된 컨텍스트를 가져 메인 세션이 오염되지 않습니다.

"3개 마이크로서비스의 로그를 동시에 각각 분석하고 비교 리포트 만들어줘"

→ Claude가 3개 Subagent 병렬 실행
→ 결과만 모아 최종 리포트 생성
→ 메인 세션 컨텍스트에는 요약만 누적

Planning Mode

복잡한 작업 전에 Claude가 먼저 계획을 세우고 승인을 받은 뒤 실행합니다. Shift + Tab으로 토글합니다 (슬래시 명령어가 아닌 키보드 단축키).

핵심 정리

오늘 다룬 것

  • CLAUDE.md — 프로젝트 구조·컨벤션 한 번 정의, 매 대화 자동 로드
  • Context Engineering — /clear · /compact · 캐싱 · 범위 지정
  • Bash / Read / Edit / Write / Grep — 명령 실행, 파일 조작
  • WebFetch / WebSearch — API 문서 조회, 최신 정보 검색
  • MCP — DB·사내 API 연결 (지연 로딩)
  • Skills — 반복 패턴을 /명령어로, global·project 2단계 스코프
  • Memory — user·project·feedback·reference 4종, 영속 저장
  • Hooks — Pre/PostToolUse 이벤트 기반 자동화
  • Subagents — 병렬 전문 에이전트, 독립 컨텍스트
