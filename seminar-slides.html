04
CLAUDE.md

CLAUDE.md란?

Claude Code가 시작될 때 자동으로 읽는 프로젝트 설명 파일.

  • 매번 "우리 프로젝트는 이런 구조야..."를 반복하지 않아도 됨
  • 팀원 모두가 동일한 컨텍스트로 Claude 사용 가능
  • 코드베이스 + 도메인 지식을 한 곳에 집약
프로젝트 루트에 두면 자동 로드. 하위 폴더에 추가로 둘 수도 있음.
# SSD Performance Lab

## 프로젝트 개요
NVMe SSD 성능 평가 자동화 도구.
FIO 기반 벤치마크 실행 및 결과 분석.

## 폴더 구조
bench/      - FIO job 파일
results/    - CSV 측정 결과
scripts/    - 분석 파이썬 스크립트
reports/    - 자동 생성 리포트

## 주요 지표
- Sequential R/W: MB/s
- Random 4K IOPS
- Latency (us): p50 / p99

## 빌드/실행
python scripts/run_bench.py --preset nvme
05
CLAUDE.md

잘 쓴 CLAUDE.md 구조

🗂 프로젝트 개요

무엇을 만드는 곳인지, 핵심 도메인 용어 정의. SSD 파트라면 FIO, IOPS, WAF 같은 용어를 여기서 정의.

📁 코드 구조

폴더별 역할 설명. Claude가 "어디를 건드려야 하는지" 탐색 없이 바로 알게 됨.

⚡ 빌드 & 실행

자주 쓰는 명령어. Claude가 직접 실행해서 결과를 확인하는 데 쓰임.

📐 코딩 컨벤션

변수명 규칙, 단위 표기 방식 등. 생성 코드가 팀 스타일을 자동으로 따르게 됨.

🚫 하지 말아야 할 것

"절대 raw device 직접 쓰지 말 것" 같은 도메인 금기사항. Claude가 실수로 위험한 명령을 내리지 않게 방어.

🔗 외부 참고

내부 위키, Jira 링크, 장비 스펙 문서 위치. Claude가 모르면 어디서 찾아야 하는지 알게 됨.

Context Engineering

컨텍스트 윈도우 — 무엇이 들어있나?

Claude는 응답을 생성할 때 컨텍스트 윈도우에 있는 모든 내용을 매번 처음부터 읽습니다. 컨텍스트가 클수록 느리고 비쌉니다.

컨텍스트 윈도우 구성 (응답 1회 기준)
① CLAUDE.md 자동 로드 · 고정
② MCP 도구 스키마 항상 점유 · 쓰든 안 쓰든
③ 대화 히스토리 누적 · 계속 증가
④ 도구 실행 결과 Bash 출력, 파일 읽기 결과
⑤ 현재 요청 지금 내가 입력한 메시지
파일 전체를 읽거나 Bash 출력이 길면 ④가 급격히 팽창합니다.

컨텍스트 크기와 비용

CLAUDE.md + MCP 스키마만저비용 ✓
+ 대화 10턴 + 파일 일부 읽기중간
+ 로그 파일 전체 붙여넣기고비용 ✗

컨텍스트 확인 명령어

  • /cost — 현재 대화에서 쓴 토큰 · 비용 확인
  • /clear — 히스토리 전체 초기화 (③④ 제거)
  • /compact — 히스토리를 요약으로 압축
Sonnet 기준 컨텍스트 한도: 200K 토큰 ≈ 약 15만 단어. 큰 것 같지만 대용량 로그 몇 개면 금방 찹니다.
07
Context Engineering

Context Engineering — 잘 쓰는 전략

컨텍스트는 한정 자원입니다. 무엇을 넣을지 의도적으로 설계하는 것이 Context Engineering입니다.

① 안정적 컨텍스트 분리

자주 바뀌지 않는 정보는 CLAUDE.md에 고정.
대화 히스토리에 반복해서 쓰지 않는다.

✗ 매번: "우리 프로젝트는 FIO 기반이고…"
✓ CLAUDE.md에 한 번 → 자동 로드

② 필요한 것만 읽기

파일 전체 대신 필요한 범위만 Read. Grep으로 관련 라인만 추출.

✗ Read("bench_10GB.csv")
✓ Read("bench.csv", offset=0, limit=100)
✓ Grep("p99", "results/")

③ 작업 단위로 초기화

무관한 작업 전환 시 /clear. 긴 작업 중간엔 /compact.

벤치마크 분석 끝 → /clear
→ 스크립트 작업 새로 시작
(이전 분석 결과가 컨텍스트 낭비)

④ 구체적인 요청

모호한 요청은 Claude가 길게 추측하며 응답 → 히스토리 팽창.

✗ "이 결과 분석해줘"
✓ "p99 레이턴시만 전주 대비
   % 변화로 한 줄 요약"

⑤ 프롬프트 캐싱 활용

Claude는 반복적으로 등장하는 컨텍스트 앞부분을 캐싱합니다. CLAUDE.md처럼 고정된 내용은 두 번째 읽기부터 비용이 크게 줄어듭니다.

CLAUDE.md → 캐시됨 (90% 절감)
대화 히스토리 → 매번 새로 읽음

⑥ MCP 서버 최소화

등록된 MCP 서버의 tool schema는 항상 컨텍스트에 상주. 안 쓰는 서버는 등록하지 않는다.

MCP 서버 1개 ≈ 수백~수천 토큰
10개 등록 → 항상 몇 천 토큰 낭비

Tools · 파일

파일 도구 — Read · Edit · Write · Grep · Glob

Read — 범위 지정 읽기

# 전체 읽기 (토큰 낭비)
Read("results/bench_10GB.csv")

# 범위 지정 (절약)
Read("results/bench_10GB.csv",
     offset=500, limit=50)

Edit — 정밀 수정

# 파일 전체 말고 해당 부분만 교체
Edit("scripts/parse.py",
  old="latency_us / 1000",
  new="latency_us / 1000.0")

Write — 새 파일 생성

# 리포트, 스크립트, 설정 파일 생성
Write("reports/jan_summary.md",
      content="...")

Grep — 코드/로그 검색

# 로그에서 패턴 검색
Grep("nvme.*timeout",
     path="logs/",
     output="context")

# 코드에서 함수 찾기
Grep("def calc_iops",
     path="scripts/")

Glob / LS — 파일 탐색

# 특정 패턴 파일 전부 찾기
Glob("results/**/*.json")

# 디렉토리 구조 확인
LS("bench/")
Claude는 보통 Glob으로 파일을 먼저 탐색하고, Read로 내용을 확인한 뒤, Edit/Write로 수정합니다. 이 순서가 자동으로 이뤄집니다.

Tools · Web

웹 도구 — WebFetch · WebSearch

WebFetch — URL 직접 읽기

특정 URL의 내용을 가져와서 Claude가 직접 분석합니다.

# 데이터시트 PDF URL 읽기
WebFetch("https://nvme.org/spec/...")

# 사내 위키 페이지 읽기
WebFetch("https://wiki.internal/ssd-spec")

# GitHub 이슈/PR 내용 가져오기
WebFetch("https://github.com/.../issues/42")
공개 데이터시트나 스펙 문서 URL을 주면 Claude가 직접 읽고 분석 가능

WebSearch — 실시간 검색

Claude 학습 데이터에 없는 최신 정보를 실시간으로 검색합니다.

# 최신 NVMe 스펙 검색
WebSearch(
  "NVMe 2.0 PCIe 5.0 latency spec")

# 드라이버 버그 검색
WebSearch(
  "linux nvme driver timeout 6.8 kernel")
사내 기밀 정보를 WebSearch 쿼리에 포함하지 말 것. 쿼리는 외부 검색엔진으로 전송됨.

언제 쓰나

  • 최신 드라이버 / 펌웨어 릴리즈 노트 확인
  • 경쟁사 벤치마크 결과 수집
  • NVMe / SATA 표준 문서 참조

Tools · Bash

Bash — SSD 분석 실전 패턴

패턴 1: 벤치마크 실행 후 즉시 분석

# Claude에게 이렇게 요청
"seq_read.fio 실행하고 결과에서
p99 레이턴시만 뽑아줘"

# Claude가 내부적으로
$ fio seq_read.fio --output-format=json
$ python parse_p99.py result.json

패턴 2: 로그에서 오류 패턴 추출

# Claude에게 이렇게 요청
"이번 주 dmesg 로그에서
NVMe timeout 발생 패턴 찾아줘"

# Claude가 내부적으로
$ grep -n "nvme.*timeout" /var/log/dmesg \
  | tail -100

permission 설정

Claude가 실행하려 할 때 허용 여부를 묻습니다. 자주 쓰는 명령은 .claude/settings.json에 등록해두면 매번 승인 불필요.

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(fio:*)",
      "Bash(python:*)",
      "Bash(grep:*)"
    ]
  }
}
rm, dd, fdisk 같은 위험 명령은 절대 allow 목록에 넣지 말 것

Tools · MCP

MCP — 외부 시스템 연결

MCP(Model Context Protocol)는 Claude가 외부 시스템을 도구로 쓸 수 있게 하는 표준 프로토콜입니다.

  • MongoDB · PostgreSQL · MySQL 쿼리
  • 사내 REST API 호출
  • Jira / Confluence 조회
  • 계측 장비 제어 (커스텀 서버)
공식 MCP 서버 목록: github.com/modelcontextprotocol/servers

등록 방법

// ~/.claude.json (글로벌 설정)
{
  "mcpServers": {
    "mongo-bench": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-mongodb"],
      "env": {
        "MONGODB_URI": "mongodb://localhost:27017/bench"
      }
    }
  }
}

MongoDB MCP 실전 예시

벤치마크 결과를 MongoDB에 저장해두면 Claude가 직접 쿼리해서 분석 가능.

# 이렇게 요청하면
"지난 달 QD32 랜덤 4K Write에서
IOPS가 가장 낮았던 SSD 모델 3개
알려줘"

# Claude가 MongoDB에 직접 쿼리
db.bench_results.aggregate([
  { $match: {
      pattern: "randwrite",
      bs: "4k", qd: 32,
      date: { $gte: "2026-05-01" }
  }},
  { $group: {
      _id: "$model",
      avg_iops: { $avg: "$iops" }
  }},
  { $sort: { avg_iops: 1 }},
  { $limit: 3 }
])
SQL을 몰라도 됩니다. 자연어로 물으면 Claude가 쿼리를 직접 작성하고 실행합니다.
MCP는 항상 컨텍스트를 차지합니다. 등록된 MCP 서버의 도구 목록(tool schema)은 대화를 시작할 때부터 컨텍스트에 자동으로 로드됩니다. 서버를 많이 등록할수록 매 응답마다 더 많은 토큰을 소비합니다. 실제로 쓰지 않는 MCP 서버는 등록하지 마세요.

Skills — 나만의 슬래시 명령어

자주 쓰는 프롬프트 패턴을 /명령어 하나로 등록하는 기능.
마크다운 파일 하나로 팀 전체가 쓸 수 있는 명령어가 만들어집니다.

적용 범위 (Scope)

🌍 Global
~/.claude/skills/
모든 프로젝트에서 사용 가능. 개인 공통 명령어.
👤 User
~/.claude/skills/
프로젝트명/
특정 프로젝트에서만 개인이 쓰는 명령어.
📁 Project
.claude/skills/
팀 공유. git에 커밋하면 팀원 모두가 동일한 명령어 사용.
예시 1: /bench-report results/jan.json
→ FIO JSON을 받아 표준 성능 리포트 자동 생성 (Project 스코프 → 팀 공유)
예시 2: /compare-run jan.json feb.json
→ 두 벤치마크 결과 비교, 회귀 감지 (Project 스코프)
Skills는 CLAUDE.md와 달리 능동적으로 호출하는 것. "이 작업을 자주 한다" 싶으면 Skill로 만들어두세요.
# .claude/skills/bench-report.md
# → Project 스코프: 팀 전체 공유

---
description: FIO 결과 파일을 받아
             표준 성능 리포트 생성
---

인자로 받은 FIO JSON 파일을 읽고
아래 형식의 리포트를 작성하세요.

## 성능 요약
| 항목      | 측정값 | 목표치 | 판정 |
|-----------|--------|--------|------|
| Seq Read  | MB/s   | 7000   | ✓/✗  |
| Rand 4K R | IOPS   | 1.5M   | ✓/✗  |
| p99 Lat   | us     | 200    | ✓/✗  |

규칙:
- Sequential R/W 단위는 MB/s
- Random은 IOPS
- Latency는 p50 / p99 모두 표시
- WAF > 3.0이면 ⚠️ 경고 추가
- 목표치 미달 항목은 원인 추정 작성

Memory

Memory — 대화를 넘어서 기억하기

Claude는 기본적으로 대화가 끝나면 모든 것을 잊습니다.
메모리 시스템은 중요한 사실을 파일로 영구 저장해서, 다음 대화 시작 때 자동으로 불러옵니다.

자동 메모리 (Auto)

Claude가 대화 중 중요하다고 판단한 내용을 .claude/memory/에 마크다운 파일로 직접 저장합니다. 사용자가 아무것도 안 해도 됩니다.

수동 메모리 (Manual)

"이거 기억해줘" 라고 말하거나 /remember 명령어를 사용합니다.

"우리 팀은 p99 레이턴시 기준이
200us야, 기억해줘"

→ Claude가 memory 파일에 저장
→ 다음 대화에서 자동으로 적용
팀 공유: .claude/memory/를 git에 커밋하면 팀원 전체가 같은 컨텍스트 공유 가능

메모리 종류 4가지

user — 나에 대한 정보

"SSD 성능 파트, Python 사용, 리포트는 한국어로"
→ 매번 소개 없이 바로 맞춤 응답

project — 현재 작업 상태

"NVMe Gen5 TLC 검증 중, 목표 IOPS 1.5M, 마감 7/31"
→ "지금 뭐 하고 있는지" 다시 설명 불필요

feedback — 선호/비선호 방식

"단위 반드시 표기 (MB not M), 표는 항상 markdown 형식"
→ 같은 피드백을 반복할 필요 없음

reference — 외부 참조

"버그는 Jira BENCH 프로젝트에, 스펙은 내부 위키 참고"
→ 어디서 찾아야 하는지 알게 됨

CLAUDE.md vs Memory

CLAUDE.md와 Memory, 무엇이 다른가?

📄 CLAUDE.md

  • 사람이 직접 작성하는 파일
  • 프로젝트 전체에 대한 고정 정보
  • git에 커밋 → 팀 전체 공유
  • 바뀌지 않는 것: 구조, 컨벤션, 실행 명령
  • Claude가 수정하지 않음
예: "이 프로젝트는 FIO 기반 벤치마크 툴이며, results/ 폴더에 CSV로 저장됩니다"
vs

🧠 Memory

  • Claude가 자동으로 저장하는 파일
  • 나(사용자)에 대한 가변적 정보
  • 대화마다 업데이트됨
  • 바뀌는 것: 현재 작업, 선호 방식, 피드백
  • Claude가 직접 쓰고 수정
예: "이 사용자는 p99 기준 200us를 목표로 하며, 단위를 반드시 명시하길 원함"
한 줄 요약: CLAUDE.md = "우리 프로젝트는 이런 곳이야" (팀이 Claude에게)  |  Memory = "너(Claude)가 나에 대해 배운 것" (Claude가 스스로 기록)

Hooks

Hooks — 이벤트 기반 자동화

Claude가 특정 행동을 할 때 자동으로 실행되는 쉘 스크립트입니다.
"Claude가 파일을 수정하면 자동으로 린터 실행" 같은 자동화가 가능합니다.

주요 Hook 이벤트

PreToolUse
도구 실행 직전 (차단 가능)
PostToolUse
도구 실행 직후
Stop
Claude가 응답 완료 시
Notification
알림 발생 시
Hooks는 Claude가 직접 실행합니다. 승인 없이 자동으로 동작하므로 신중하게 작성하세요.

SSD 파트 활용 예시

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "python -m pytest scripts/ -q"
      }]
    }],
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "scripts/check_dangerous_cmd.sh"
      }]
    }]
  }
}
→ 스크립트 수정 시 자동 테스트 실행
→ 위험 명령(dd, fdisk) 사전 차단

Subagents & Planning

Subagents · Planning Mode · Extended Thinking

Subagents — 전문화된 병렬 에이전트

복잡한 작업을 여러 전문 에이전트에 병렬로 위임합니다. 각 에이전트는 독립된 컨텍스트를 가져 서로 간섭하지 않습니다.

# 예: 벤치마크 다중 분석
"3개 드라이브의 FIO 결과를
 동시에 각각 분석하고
 비교 리포트 만들어줘"

→ Claude가 3개 Subagent 생성
→ 각자 독립적으로 분석
→ 결과를 모아 최종 비교
Subagent는 메인 컨텍스트를 오염시키지 않습니다. 대용량 분석에 적합합니다.

Planning Mode — 실행 전 계획 수립

복잡한 작업 전에 Claude가 먼저 계획을 세우고 승인을 받은 뒤 실행합니다.

# 활성화
/plan

# 또는 요청에 포함
"실행 전에 먼저 계획을
 보여줘, 확인 후 진행할게"

→ 대규모 리팩토링, 여러 파일 수정 전 검토에 유용

Extended Thinking

Claude가 응답 전 더 깊이 추론하도록 합니다. 복잡한 성능 병목 원인 분석에 효과적입니다.

# 토글 단축키
Option + T  (Mac)
Alt + T     (Windows)

# 또는 요청에 포함
"깊이 생각해서 이 레이턴시
 스파이크의 원인을 분석해줘"
Extended Thinking은 응답이 느리고 비용이 높습니다. 단순 작업엔 사용하지 마세요.

CLI

CLI 활용 — 파이프라인 & 자동화

Claude Code는 대화형 모드 외에 CLI 비대화형 모드로 쓸 수 있습니다. 스크립트, CI/CD, 쉘 파이프라인에 통합할 때 사용합니다.

# 대화형 (기본)
$ claude

# 비대화형 (스크립트용)
$ claude -p "이 코드 리뷰해줘"

# 파이프라인 — stdin 읽기
$ cat bench_result.json \
  | claude -p "p99 레이턴시 요약"

# 파일 직접 전달
$ claude -p "이 로그 분석" \
  --file dmesg.log

# 세션 재개
$ claude -r "bench-session" \
  "어제 분석 이어서 해줘"

SSD 자동화 파이프라인 예시

#!/bin/bash
# 매일 밤 자동 벤치마크 + 분석 + 리포트

# 1. FIO 실행
fio bench/nightly.fio \
  --output-format=json \
  > results/$(date +%F).json

# 2. Claude로 자동 분석
cat results/$(date +%F).json \
  | claude -p "전일 대비 성능 비교
    이상값 감지, 한국어 요약" \
  > reports/$(date +%F).md

# 3. 슬랙 알림 (선택)
cat reports/$(date +%F).md \
  | slack-notify --channel bench
-p 플래그(print)는 stdout으로만 출력하고 종료합니다. 크론잡, 쉘 스크립트에 통합하기 좋습니다.
유용한 플래그
-p "prompt" 출력만 하고 종료
-r "name" 이름 붙인 세션 재개
--file 파일 첨부
--model 모델 지정
