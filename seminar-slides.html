Agent

Claude Code는 왜 "Agent"인가?

일반 LLM은 텍스트를 받아 텍스트를 돌려줍니다.
Agent는 목표를 받아 스스로 계획하고, 도구를 선택해 실행하고, 결과를 보고 다시 판단합니다.

일반 LLM vs Agent
💬 일반 LLM (ChatGPT 등)
질문 → 텍스트 답변
파일을 직접 읽지 못함
명령을 직접 실행 못함
한 번의 응답으로 끝
🤖 Agent (Claude Code)
목표 → 계획 → 도구 실행
파일을 읽고 수정
명령어를 직접 실행
결과를 보고 다음 행동 결정
"로그에서 에러 패턴 분석해줘"
→ Glob으로 파일 탐색 → Read로 로그 읽기 → Bash로 grep 실행 → 결과 분석 → 리포트 Write
이 모든 과정을 Claude가 스스로 판단하며 진행합니다.

Agent를 가능하게 하는 것: Tools

도구(Tool)는 Claude가 세상과 상호작용하는 손입니다.
도구 없이는 LLM, 도구와 함께하면 Agent입니다.

읽기

WebFetch · WebSearch
쓰기

Claude Code의 도구들

⚡ Bash

쉘 명령 직접 실행.

🔍 Grep

파일 내 텍스트 검색.
정규식 지원, 라인 번호 반환

패턴으로 파일 찾기.
디렉토리 목록 조회

DB, API, 장비 제어

Claude는 이 도구들을 스스로 조합해서 사용합니다. "로그 분석해줘" 한 마디에 Glob → Read → Bash(python) 순서로 자동 실행합니다.

Tools · 파일

파일 도구 — Read · Edit · Write · Grep · Glob

Read — 범위 지정 읽기

# 전체 읽기 (토큰 낭비)
Read("logs/app.log")

# 범위 지정 (절약)
Read("logs/app.log",
     offset=500, limit=50)

Edit — 정밀 수정

# 파일 전체 말고 해당 부분만 교체
Edit("scripts/parse.py",
  old="response_ms / 1000",
  new="response_ms / 1000.0")

Write — 새 파일 생성

# 리포트, 스크립트, 설정 파일 생성
Write("reports/jan_summary.md",
      content="...")

Grep — 코드/로그 검색

# 로그에서 패턴 검색
Grep("ERROR\|WARN",
     path="logs/",
     output="context")

# 코드에서 함수 찾기
Grep("def parse_response",
     path="scripts/")

Glob / LS — 파일 탐색

# 특정 패턴 파일 전부 찾기
Glob("logs/**/*.json")

# 디렉토리 구조 확인
LS("tests/")
Claude는 보통 Glob으로 파일을 먼저 탐색하고, Read로 내용을 확인한 뒤, Edit/Write로 수정합니다. 이 순서가 자동으로 이뤄집니다.

Tools · Web

웹 도구 — WebFetch · WebSearch

WebFetch — URL 직접 읽기

특정 URL의 내용을 가져와서 Claude가 직접 분석합니다.

# 라이브러리 공식 문서 읽기
WebFetch("https://docs.example.com/api/...")

# 사내 위키 페이지 읽기
WebFetch("https://wiki.internal/api-spec")

# GitHub 이슈/PR 내용 가져오기
WebFetch("https://github.com/.../issues/42")
공개 API 문서나 라이브러리 문서 URL을 주면 Claude가 직접 읽고 분석 가능

WebSearch — 실시간 검색

Claude 학습 데이터에 없는 최신 정보를 실시간으로 검색합니다.

# 최신 라이브러리 변경사항 검색
WebSearch(
  "React 19 breaking changes migration")

# 프레임워크 버그 검색
WebSearch(
  "Next.js 15 server action error handling")
사내 기밀 정보를 WebSearch 쿼리에 포함하지 말 것. 쿼리는 외부 검색엔진으로 전송됨.

언제 쓰나

  • 최신 라이브러리 / 프레임워크 릴리즈 노트 확인
  • 오픈소스 이슈 트래커 검색
  • API / 프로토콜 표준 문서 참조

Tools · Bash

Bash — 개발 실전 패턴

패턴 1: 테스트 실행 후 즉시 분석

# Claude에게 이렇게 요청
"python run_tests.py 실행하고 결과에서
실패한 테스트만 뽑아줘"

# Claude가 내부적으로
$ python run_tests.py --output-format=json
$ python parse_logs.py result.json

패턴 2: 로그에서 오류 패턴 추출

# Claude에게 이렇게 요청
"이번 주 app.log에서
ERROR, WARN 발생 패턴 찾아줘"

# Claude가 내부적으로
$ grep -n "ERROR\|WARN" /var/log/app.log \
  | tail -100

permission 설정

Claude가 실행하려 할 때 허용 여부를 묻습니다. 자주 쓰는 명령은 .claude/settings.json에 등록해두면 매번 승인 불필요.

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(python:*)",
      "Bash(npm:*)",
      "Bash(grep:*)"
    ]
  }
}
rm, curl --data, 프로덕션 배포 명령은 절대 allow 목록에 넣지 말 것

Tools · MCP

MCP — 외부 시스템 연결

MCP(Model Context Protocol)는 Claude가 외부 시스템을 도구로 쓸 수 있게 하는 표준 프로토콜입니다.

  • MongoDB · PostgreSQL · MySQL 쿼리
  • 사내 REST API 호출
  • Jira / Confluence 조회
  • 사내 서비스 제어 (커스텀 서버)

등록 방법

// ~/.claude.json
{
  "mcpServers": {
    "mongo": {
      "command": "npx",
      "args": ["-y", "@mcp/server-mongodb"],
      "env": { "MONGODB_URI": "mongodb://..." }
    }
  }
}
항상 컨텍스트 점유. 등록된 MCP 서버의 tool schema는 쓰든 안 쓰든 매 응답마다 로드됩니다. 안 쓰는 서버는 등록하지 마세요.

실전 예시 — 자연어 → DB 쿼리

# 요청
"지난달 응답시간 가장 높은
엔드포인트 TOP 3 알려줘"

# Claude가 자동 생성·실행
db.api_logs.aggregate([
  { $group: {
      _id: "$endpoint",
      avg_ms: { $avg: "$response_ms" }
  }},
  { $sort: { avg_ms: -1 }},
  { $limit: 3 }
])
쿼리 문법을 몰라도 됩니다. 자연어로 물으면 Claude가 직접 작성하고 실행합니다.
공식 서버 목록: github.com/modelcontextprotocol/servers

CLAUDE.md

CLAUDE.md란?

Claude Code가 시작될 때 자동으로 읽는 프로젝트 설명 파일.

  • 매번 "우리 프로젝트는 이런 구조야..."를 반복하지 않아도 됨
  • 팀원 모두가 동일한 컨텍스트로 Claude 사용 가능
  • 코드베이스 + 도메인 지식을 한 곳에 집약
누가 수정하나?
👤 사람이 직접 작성
프로젝트 초기 설정, 컨벤션·규칙 정의, 도메인 지식 입력
🤖 Claude가 직접 수정
"CLAUDE.md에 이 컨벤션 추가해줘" 요청 시, 또는 작업 중 새로운 패턴을 발견하면 Claude가 스스로 Edit 도구로 업데이트 가능
프로젝트 루트에 두면 자동 로드. 하위 폴더에 추가로 둘 수도 있음.
# Service Monitor

## 프로젝트 개요
웹 서비스 모니터링 자동화 도구.
API 로그 수집 및 이상 감지 분석.

## 폴더 구조
tests/      - 테스트 스크립트
logs/       - 수집된 로그 파일
scripts/    - 분석 파이썬 스크립트
reports/    - 자동 생성 리포트

## 주요 지표
- API 응답시간: ms
- RPS (Requests Per Second)
- 에러율: 4xx / 5xx

## 빌드/실행
python run_tests.py --env staging

CLAUDE.md

왜 Markdown으로 쓰나?

Claude는 인터넷의 방대한 Markdown 문서로 학습됐습니다. Markdown 구조 자체가 Claude에게 의미 신호가 됩니다.

Markdown 요소별 Claude의 해석
## 제목 "이 섹션은 X에 관한 정보다" — 구조를 파악하고 관련 내용만 참조
- 항목 나열된 규칙 · 옵션으로 인식 — 빠짐없이 따라야 할 목록
```code``` 실행 가능한 명령어 · 코드 — 그대로 쓰거나 실행할 것
**굵게** 중요 키워드 · 강조 — 높은 가중치로 처리
일반 텍스트도 동작하지만, Markdown을 쓰면 Claude가 파싱 오류 없이 구조를 즉시 인식합니다.

같은 내용, 다른 결과

✗ 일반 텍스트 (구조 없음)
프로젝트 개요: 웹 서비스 모니터링 도구임. 폴더 구조는 tests가 테스트 scripts가 분석 reports가 리포트임. 빌드는 python run_tests.py로 함. main 브랜치에만 머지할 것.
✓ Markdown (구조 명확)
## 프로젝트 개요
웹 서비스 모니터링 도구.

## 폴더 구조
- tests/   → 테스트 스크립트
- scripts/ → 분석 코드
- reports/ → 자동 생성 리포트

## 빌드
```python run_tests.py```

## 규칙
- main 브랜치에만 머지
Claude도 읽고, 팀원도 읽는 파일.
Markdown은 두 독자 모두를 위한 선택입니다.

CLAUDE.md

잘 쓴 CLAUDE.md 구조

🗂 프로젝트 개요

무엇을 만드는 곳인지, 핵심 도메인 용어 정의. 팀 도메인 약어와 개념을 여기서 정의해두면 Claude가 즉시 이해합니다.

📁 코드 구조

폴더별 역할 설명. Claude가 "어디를 건드려야 하는지" 탐색 없이 바로 알게 됨.

⚡ 빌드 & 실행

자주 쓰는 명령어. Claude가 직접 실행해서 결과를 확인하는 데 쓰임.

📐 코딩 컨벤션

변수명 규칙, 단위 표기 방식 등. 생성 코드가 팀 스타일을 자동으로 따르게 됨.

🚫 하지 말아야 할 것

"프로덕션 DB에 직접 쓰지 말 것" 같은 안전 규칙. Claude가 실수로 위험한 명령을 내리지 않게 방어.

🔗 외부 참고

내부 위키, Jira 링크, 장비 스펙 문서 위치. Claude가 모르면 어디서 찾아야 하는지 알게 됨.

Context Engineering

Context란 무엇인가?

Claude에게는 기억이 없습니다

Claude는 무상태(stateless) 모델입니다. 대화가 끝나면 모든 것을 잊습니다. 어제 나눈 대화도, 방금 분석한 파일도 기억하지 않습니다.

새 대화를 시작하면
🤖
완전한 백지 상태로 시작합니다.
어떤 프로젝트인지, 이전에 무슨 작업을 했는지 전혀 모름

Context = Claude의 작업 메모리

Claude가 응답을 생성하기 직전, "지금 이 순간 알고 있는 모든 것"이 Context입니다. 대화창에 보이는 내용 + 시스템이 자동으로 넣어준 내용 전부.

컴퓨터의 RAM과 같습니다. 전원을 끄면(=대화 종료) 사라지고, 켜는 동안(=한 대화 안에서)만 유지됩니다.

Context가 중요한 이유

Context에 있는 것 vs 없는 것
✓ Context 안에 있으면
Claude가 알고 있음
답할 수 있음
실행에 반영됨
✗ Context 밖에 있으면
Claude가 모름
추측하거나 틀림
반영되지 않음
예시
✗ "API 에러 왜 났어?"
→ 프로젝트 정보, 에러 내용이 Context에 없음. Claude는 모름. ✓ "API 에러 왜 났어?"
→ CLAUDE.md + logs/error.log가 Context에 있음. Claude가 분석 가능.
Context Engineering = 무엇을 Context에 넣을지 의도적으로 설계하는 것

Context Engineering

컨텍스트 윈도우 — 무엇이 들어있나?

Claude는 응답을 생성할 때 컨텍스트 윈도우에 있는 모든 내용을 매번 처음부터 읽습니다. 컨텍스트가 클수록 느리고 비쌉니다.

컨텍스트 윈도우 구성 (응답 1회 기준)
① CLAUDE.md 자동 로드 · 고정
② MCP 도구 스키마 항상 점유 · 쓰든 안 쓰든
③ 대화 히스토리 누적 · 계속 증가
④ 도구 실행 결과 Bash 출력, 파일 읽기 결과
⑤ 현재 요청 지금 내가 입력한 메시지
파일 전체를 읽거나 Bash 출력이 길면 ④가 급격히 팽창합니다.

컨텍스트 크기와 비용

CLAUDE.md + MCP 스키마만저비용 ✓
+ 대화 10턴 + 파일 일부 읽기중간
+ 로그 파일 전체 붙여넣기고비용 ✗

컨텍스트 확인 명령어

  • /cost — 현재 대화에서 쓴 토큰 · 비용 확인
  • /clear — 히스토리 전체 초기화 (③④ 제거)
  • /compact — 히스토리를 요약으로 압축
모델별 컨텍스트 한도: Sonnet 200K / Opus 1M 토큰. 큰 것 같지만 대용량 로그 몇 개면 금방 찹니다.

Context Engineering

Context Engineering — 잘 쓰는 전략

컨텍스트는 한정 자원입니다. 무엇을 넣을지 의도적으로 설계하는 것이 Context Engineering입니다.

① 안정적 컨텍스트 분리

자주 바뀌지 않는 정보는 CLAUDE.md에 고정.
대화 히스토리에 반복해서 쓰지 않는다.

✗ 매번: "우리 프로젝트는 이런 구조이고…"
✓ CLAUDE.md에 한 번 → 자동 로드

② 필요한 것만 읽기

파일 전체 대신 필요한 범위만 Read. Grep으로 관련 라인만 추출.

✗ Read("app.log")
✓ Read("app.log", offset=0, limit=100)
✓ Grep("ERROR", "logs/")

③ 작업 단위로 초기화

무관한 작업 전환 시 /clear. 긴 작업 중간엔 /compact.

로그 분석 끝 → /clear
→ 코드 작업 새로 시작
(이전 작업 결과가 컨텍스트 낭비)

④ 구체적인 요청

모호한 요청은 Claude가 길게 추측하며 응답 → 히스토리 팽창.

✗ "이 로그 분석해줘"
✓ "API 응답시간만 전주 대비
   % 변화로 한 줄 요약"

⑤ 프롬프트 캐싱 활용

Claude는 반복적으로 등장하는 컨텍스트 앞부분을 캐싱합니다. CLAUDE.md처럼 고정된 내용은 두 번째 읽기부터 비용이 크게 줄어듭니다.

CLAUDE.md → 캐시됨 (90% 절감)
대화 히스토리 → 매번 새로 읽음

⑥ MCP 서버 최소화

등록된 MCP 서버의 tool schema는 항상 컨텍스트에 상주. 안 쓰는 서버는 등록하지 않는다.

MCP 서버 1개 ≈ 수백~수천 토큰
10개 등록 → 항상 몇 천 토큰 낭비
-
Context → Skills

그래도 반복되는 것이 있다

CLAUDE.md로 프로젝트 지식은 해결됐습니다. 하지만 자주 반복하는 요청 패턴은 여전히 매번 타이핑해야 합니다.

해결 안 된 문제

매 스프린트마다 반복하는 요청
"지난주 API 에러 로그 분석해서
  에러 유형별로 집계하고,
  개선이 필요한 엔드포인트 TOP 3 뽑아줘.
  결과는 마크다운 표로."
이걸 매번 타이핑? 팀원마다 조금씩 다르게 요청해서 일관성도 없음.

더 큰 문제

  • 프롬프트가 길수록 실수·누락이 생김
  • 노하우가 개인에게 묶임 (공유 불가)
  • CLAUDE.md에 넣기엔 너무 긴 절차

Skills의 등장

반복 프롬프트를 /명령어 하나로 등록합니다.
팀 전체가 같은 방식으로 Claude를 씁니다.

이제 이렇게만 입력
/api-report
Skills는 단순 단축키가 아닙니다. 긴 절차, 판단 기준, 출력 형식까지 담은 팀의 AI 사용 표준입니다.

Harness Engineering

Harness Engineering이란?

핵심 관점

Claude Code 자체가 이미 잘 구성된 harness입니다.

Anthropic이 Claude LLM 위에 도구(Tools), 승인 시스템, 파일 접근, MCP 연결을 얹어 만든 것이 Claude Code — 즉, harness의 첫 번째 레이어입니다.

우리가 하는 것

그 위에 팀만의 harness를 추가합니다.

CLAUDE.md · Settings · Skills · Memory · Hooks로 Claude Code를 우리 프로젝트에 맞게 커스터마이징하는 것이 Harness Engineering입니다.

Skills — 슬래시 명령어 · 자동 호출

자주 쓰는 프롬프트 패턴을 등록해두는 기능. 호출 방식이 두 가지입니다.

Skill 호출 방식
👤 User-invoked
사용자가 직접 슬래시 명령어로 호출
/api-report logs/jan.json
명시적·의도적. 언제 실행되는지 사용자가 제어.
🤖 Model-invoked
Claude가 대화 맥락을 보고 자동으로 실행
"PR 리뷰해줘" 입력
→ Claude가 /code-review skill 자동 선택
Skill의 description이 판단 기준.
Model-invoked가 동작하려면 description frontmatter를 잘 써야 합니다.
Claude는 이 description을 읽고 "지금 상황에 이 Skill을 써야겠다"고 판단합니다.

적용 범위 (Scope)

🙋 Personal
~/.claude/skills/
내 모든 프로젝트에서 사용. 개인 공통 명령어.
📁 Project
.claude/skills/
팀 공유. git에 커밋하면 팀원 모두가 동일한 명령어 사용.
예시 1: /api-report logs/jan.json
→ 테스트 결과 JSON을 받아 표준 API 리포트 자동 생성 (Project 스코프 → 팀 공유)
예시 2: /code-review src/api.py
→ 지정 파일 코드 리뷰, 개선 사항 감지 (Project 스코프)
Skills는 CLAUDE.md와 달리 능동적으로 호출하는 것. "이 작업을 자주 한다" 싶으면 Skill로 만들어두세요.

Memory

Memory — 대화를 넘어서 기억하기

Claude는 기본적으로 대화가 끝나면 모든 것을 잊습니다.
메모리 시스템은 중요한 사실을 파일로 영구 저장해서, 다음 대화 시작 때 자동으로 불러옵니다.

자동 메모리 (Auto)

Claude가 대화 중 중요하다고 판단한 내용을 .claude/memory/에 마크다운 파일로 직접 저장합니다. 사용자가 아무것도 안 해도 됩니다.

수동 메모리 (Manual)

"이거 기억해줘" 라고 말하거나 /remember 명령어를 사용합니다.

"우리 팀은 API 응답시간 기준이
200ms야, 기억해줘"

→ Claude가 memory 파일에 저장
→ 다음 대화에서 자동으로 적용
팀 공유: .claude/memory/를 git에 커밋하면 팀원 전체가 같은 컨텍스트 공유 가능

메모리 종류 4가지

user — 나에 대한 정보

"백엔드 개발자, Python/TypeScript 사용, 리포트는 한국어로"
→ 매번 소개 없이 바로 맞춤 응답

project — 현재 작업 상태

"REST API 리팩토링 중, 목표 RPS 10K, 마감 7/31"
→ "지금 뭐 하고 있는지" 다시 설명 불필요

feedback — 선호/비선호 방식

"타입 반드시 명시 (ms not 숫자만), 코드는 항상 타입 힌트 포함"
→ 같은 피드백을 반복할 필요 없음

reference — 외부 참조

"버그는 Jira DEV 프로젝트에, 스펙은 내부 위키 참고"
→ 어디서 찾아야 하는지 알게 됨

CLAUDE.md vs Memory

CLAUDE.md와 Memory, 무엇이 다른가?

📄 CLAUDE.md

  • 사람이 직접 작성하는 파일
  • 프로젝트 전체에 대한 고정 정보
  • git에 커밋 → 팀 전체 공유
  • 바뀌지 않는 것: 구조, 컨벤션, 실행 명령
  • Claude가 수정하지 않음
예: "이 프로젝트는 REST API 서비스이며, logs/ 폴더에 JSON으로 저장됩니다"
vs

🧠 Memory

  • Claude가 자동으로 저장하는 파일
  • 나(사용자)에 대한 가변적 정보
  • 대화마다 업데이트됨
  • 바뀌는 것: 현재 작업, 선호 방식, 피드백
  • Claude가 직접 쓰고 수정
예: "이 사용자는 API 응답시간 기준 200ms를 목표로 하며, 타입을 반드시 명시하길 원함"
한 줄 요약: CLAUDE.md = "우리 프로젝트는 이런 곳이야" (팀이 Claude에게)  |  Memory = "너(Claude)가 나에 대해 배운 것" (Claude가 스스로 기록)

Hooks

Hooks — 이벤트 기반 자동화

Claude가 특정 행동을 할 때 자동으로 실행되는 쉘 스크립트입니다.
"Claude가 파일을 수정하면 자동으로 린터 실행" 같은 자동화가 가능합니다.

주요 Hook 이벤트

PreToolUse
도구 실행 직전 (차단 가능)
PostToolUse
도구 실행 직후
Stop
Claude가 응답 완료 시
Notification
알림 발생 시
Hooks는 Claude가 직접 실행합니다. 승인 없이 자동으로 동작하므로 신중하게 작성하세요.

활용 예시

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "python -m pytest scripts/ -q"
      }]
    }],
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "scripts/check_dangerous_cmd.sh"
      }]
    }]
  }
}
→ 스크립트 수정 시 자동 테스트 실행
→ 위험 명령(dd, fdisk) 사전 차단

Subagents & Planning

Subagents · Planning Mode

Subagents — 전문화된 병렬 에이전트

복잡한 작업을 여러 전문 에이전트에 병렬로 위임합니다. 각 에이전트는 독립된 컨텍스트를 가져 서로 간섭하지 않습니다.

# 예: 서비스 다중 분석
"3개 마이크로서비스의 로그를
 동시에 각각 분석하고
 비교 리포트 만들어줘"

→ Claude가 3개 Subagent 생성
→ 각자 독립적으로 분석
→ 결과를 모아 최종 비교
Subagent는 메인 컨텍스트를 오염시키지 않습니다. 대용량 분석에 적합합니다.

Planning Mode — 실행 전 계획 수립

복잡한 작업 전에 Claude가 먼저 계획을 세우고 승인을 받은 뒤 실행합니다.

# 활성화 — Shift+Tab 토글
Shift + Tab  ← 슬래시 명령 아님

# 또는 요청에 포함
"실행 전에 먼저 계획을
 보여줘, 확인 후 진행할게"

→ 대규모 리팩토링, 여러 파일 수정 전 검토에 유용

Summary

핵심 정리

  • CLAUDE.md — 프로젝트 구조·컨벤션 한 번 정의, 매 대화 자동 로드
  • Context Engineering — 컨텍스트 구성 의도적 설계 · /clear · /compact · 캐싱
  • Bash / Read / Edit / Write / Grep — 명령 실행, 파일 조작, 패턴 검색
  • WebFetch / WebSearch — API 문서 조회, 최신 정보 검색
  • MCP — MongoDB·DB·사내 API 연결 (항상 컨텍스트 점유 주의)
  • Skills — 반복 패턴을 /명령어로, 2단계 스코프 (personal·project)
  • Memory — user·project·feedback·reference 4종, 영속 저장
  • Hooks — 이벤트 기반 자동화, Pre/PostToolUse 스크립트
  • Subagents — 병렬 전문 에이전트, 독립 컨텍스트
  • Harness Engineering — CLAUDE.md · Settings · Skills · Memory · Hooks를 시스템으로 묶기. Claude Code 자체가 1st layer, 우리 설정이 2nd layer

오늘부터 할 수 있는 것

1️⃣ 프로젝트 루트에 CLAUDE.md 만들기
팀 구조 + 주요 명령어 + 코딩 컨벤션 기준

2️⃣ /api-report Skill 하나 만들어보기
